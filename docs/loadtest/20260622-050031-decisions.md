# Load Test Decisions

Generated from `reports/loadtest/20260622-050031`.

## Verdict

The current run is useful, but it is not a capacity limit for the whole system.
The API, VK webhook, Mini App basic paths and billing mock paths stayed fast.
The first actionable issue is in the job/worker load scenario and queue
draining, not in API CPU, Postgres CPU or Redis memory.

## Decision 1. First Bottleneck

Current first bottleneck candidate: job/worker path plus delivery backlog.

The job-worker scenario failed the HTTP error-rate threshold. Manual follow-up
showed text and image mock job creation worked, while a video job route returned
`400`. Treat the current job-worker failure as a video route/config/load-test
fixture problem first, then rerun before concluding the worker is overloaded.

## Decision 2. API Scaling

Do not scale API yet.

Reason: API-facing scenarios showed low latency and zero error rate. CPU/RAM are
also low. Scaling API before fixing job route validity and worker/delivery
throughput would hide the real queue pressure.

## Decision 3. Worker Count

Keep one worker for the next diagnostic rerun, then test two worker instances.

Reason: current worker throughput was about 0.69 terminal jobs/sec, but the run
is contaminated by the video route failure and accumulated Redis backlog. The
next valid comparison should be:

1. clean loadtest Redis/DB;
2. text-only job run;
3. image-only job run;
4. video-only mock route run;
5. mixed run with one worker;
6. mixed run with two workers.

If queue depth still grows with valid mock jobs, add worker replicas before
changing API or Redis.

## Decision 4. Queue Topology

Keep separated streams and add per-stream attention thresholds.

Existing streams already separate text, image, video, delivery, provider polling
and DLQ. The current delivery stream depth is the largest signal, so the next
work should measure whether delivery needs its own worker pool.

Recommended thresholds for the next loadtest contour:

```text
stream:jobs:text       warn > 100
stream:jobs:image      warn > 50
stream:jobs:video      warn > 20
stream:jobs:delivery   warn > 100
stream:jobs:dlq        fail > 0
```

## Decision 5. Postgres And Indexes

Do not add new indexes from this run alone.

Reason: tables are small, no locks were found, and `pg_stat_statements` is not
enabled. Sequential-scan hints are useful, but not enough to justify broad index
changes.

Next database action:

```text
enable pg_stat_statements on load/staging
rerun the same load
rank queries by total_exec_time and mean_exec_time
add only targeted indexes backed by those numbers
```

## Decision 6. Redis

Redis capacity is not the current limit.

Reason: memory is low, rejected connections are zero and DLQ is zero. The high
queue depth means producers and consumers are not balanced, not that Redis needs
to be moved or resized immediately.

Next Redis action:

- clear or recreate loadtest Redis before each official run;
- keep reporting XLEN per stream;
- add pending/idle consumer checks to the pass/fail decision.

## Decision 7. Data Services

Do not move Postgres/Redis/S3 because of this local run alone.

The code and deploy are now prepared for local/external/managed data services,
but this run does not prove that managed services are the immediate bottleneck.

Move data services out of the app VPS before serious horizontal scaling or real
large traffic, not as the first response to this report.

## Decision 8. User Limits

Keep current anti-spam and active-job protections enabled.

Synthetic VK load already created cooldown/block keys. That is expected and
means anti-spam is participating in burst protection. For throughput tests, use
unique synthetic users or a dedicated mode; for abuse tests, intentionally reuse
the same user.

Next limit work:

- verify active jobs per user under mixed job load;
- add per-modality limits if video jobs can starve text/image jobs;
- keep provider-spend limits independent from generic message limits.

## Decision 9. Provider Quotas

No provider quota decision can be made from this run.

Providers were mocked, correctly. Real FAL, Runway, APIMart, PoYo, DeepInfra or
YooKassa quota checks must be separate low-volume live-smoke tests with explicit
approval.

## Next Run Plan

Before the next load run:

1. fix the video mock route used by `tests/k6/job-worker.js` or change it to a
   route that is enabled in `APP_ENV=loadtest`;
2. start with a clean loadtest Redis/DB or reset job streams;
3. run `basic-api`, `vk-bot`, `billing-payments` to confirm baseline remains
   green;
4. run `job-worker` as text-only, image-only, video-only and mixed;
5. repeat the mixed run with two worker instances;
6. compare queue depth, worker throughput and p95/p99.

The next engineering decision should be based on that clean one-worker vs
two-worker comparison.
