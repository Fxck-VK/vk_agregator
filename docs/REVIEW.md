# REVIEW.md — Аудит безопасности и архитектуры

Ветка: `feature/vk-miniapp`. Режим: только чтение (код не менялся).
Дата: 2026-06-04. Ревьюер: senior security/architecture review.

Охват: Go-бэкенд (`internal/`, `cmd/api`, `cmd/worker`), React-фронтенд VK Mini App
(`web/miniapp`). Глубина — фактическая проверка кода по перечню задач.

Общая оценка: базовый периметр построен правильно (fail-closed в проде,
ownership-проверки, параметризованный SQL, append-only ledger, идемпотентность
денег). Главные пробелы — отсутствие rate-limiting на `/miniapp/*`, мягкая
проверка времени жизни launch-параметров и пара рассинхронов фронт↔бэк.

## 1. Безопасность (бэкенд)

### [Verdict: OK] Верификация launch-параметров VK
`internal/adapter/inbound/miniapp/sign.go:45-93`
HMAC-SHA256, фильтр `vk_*`, сортировка через `url.Values.Encode()`, сравнение
`hmac.Equal` (constant-time), base64url без паддинга. Алгоритм соответствует
спецификации VK. Пустой `VK_APP_SECRET` пропускает подпись — но это **fail-closed
в проде**: `config.Validate()` (`internal/platform/config/config.go:159-163`)
требует `VK_APP_SECRET` при `APP_ENV=production`, иначе процесс не стартует
(`cmd/api/main.go:41-44`). Dev-обход через заголовок `X-VK-User-ID`
(`handler.go:90-105`) активен только когда секрет пуст, т.е. вне прода. Корректно.

### [SEVERITY: High] Нет rate-limiting на эндпоинтах Mini App
`cmd/api/main.go:151-158`
Per-IP лимитер навешен только на `/webhooks/vk`
(`webhookLimiter.Middleware(...)`). Маршруты `/miniapp/*` (включая
`POST /miniapp/jobs`, который создаёт биллируемые Job) ничем не ограничены.
Один аутентифицированный пользователь может залить интейк потоком запросов:
нагрузка на БД/очередь, рост числа резерваций, исчерпание баланса/квот.
Баланс ограничивает стоимость, но не частоту и не объём create-нагрузки.
**Рекомендация:** обернуть `miniapp.Routes()` в тот же `ratelimit` (ключ — по
`vk_user_id` или IP), задать отдельные RPS/Burst, минимально — на `POST /jobs`.

### [SEVERITY: Medium] Время жизни launch-параметров проверяется «мягко»
`internal/adapter/inbound/miniapp/sign.go:75-86`
Проверка `vk_ts` выполняется только если параметр присутствует и парсится.
Если `vk_ts` отсутствует или нечисловой — возраст **не проверяется**, и валидно
подписанная строка живёт вечно (окно для повторного использования перехваченных
launch-параметров). При `maxAge > 0` отсутствие/битость `vk_ts` должны
трактоваться как ошибка, а не как «пропустить».
**Рекомендация:** при `maxAge > 0` требовать наличие корректного `vk_ts`
(пустой/битый → `ErrExpiredParams`).

### [Verdict: OK] Ownership-проверки и проброс vk_user_id
`handler.go:301-341` (`getJob`), `handler.go:259-299` (`listJobs`),
`handler.go:369-427` (`getArtifact`).
`vk_user_id` кладётся в контекст только после верификации; `getJob` и
`getArtifact` сравнивают `job.UserID`/`art.OwnerUserID` с пользователем и при
несовпадении отдают `404 not found` (не раскрывая существование). `listJobs`
читает строго `Jobs.ListByUser(user.ID, ...)`. Межпользовательский доступ закрыт.

### [Verdict: OK] Идемпотентность (X-Idempotency-Key)
`handler.go:218-225`
Ключ скоупится пользователем: `miniapp_job:<vk_user_id>:<clientKey>`, поэтому
один пользователь не может подменить/переиграть ключ другого. На уровне БД —
`UNIQUE` на `jobs.idempotency_key`, в оркестраторе — lookup до создания
(`joborchestrator/orchestrator.go:92-98`). Корректно.

### [Verdict: OK] SQL — параметризация
`internal/adapter/storage/postgres/job.go`, `billing.go`.
Все запросы используют `$N`-плейсхолдеры. Динамический `List`
(`job.go:149-190`) строит из фильтров только `$N`-позиции (`fmt.Sprintf("$%d")`),
значения уходят в `args`, конкатенации пользовательских данных нет. Инъекций не
обнаружено.

### [SEVERITY: Low] Сравнение ADMIN_TOKEN не константно по времени
`internal/adapter/inbound/admin/handler.go:61`
`r.Header.Get("X-Admin-Token") != h.cfg.Token` — обычное сравнение строк
(теоретический timing-side-channel). При пустом `Token` auth пропускается, но
это fail-closed в проде (`config.go:153-155`). Риск низкий, токен внутренний.
**Рекомендация:** `hmac.Equal`/`subtle.ConstantTimeCompare`.

### [SEVERITY: Low] Нет верхней границы длины prompt и CORS-политики
`handler.go:190-203`
Тело ограничено 64 KiB (`io.LimitReader(r.Body, 64<<10)`), `operation`
валидируется по белому списку, пустой `prompt` отклоняется — это хорошо. Но
отдельного лимита длины `prompt` (в пределах 64 KiB) нет, и CORS-заголовки нигде
не выставляются.
**Рекомендация:** при доступе строго через same-origin proxy CORS не нужен —
**требует уточнения** модели развёртывания; при прямом доступе добавить
строгий `Access-Control-Allow-Origin`. Доп. лимит длины prompt — опционально.

## 2. Безопасность (фронтенд, web/miniapp)

### [Verdict: OK] X-Launch-Params, отсутствие eval/innerHTML, src через artifactUrl
- `src/api/client.ts:51` — `LAUNCH_PARAMS` берётся **только** из
  `window.location.search`, не из localStorage/стора.
- Нет `dangerouslySetInnerHTML`/`eval`/`new Function`; текст рендерится как
  `{msg.text}` (React экранирует) — `src/chat/MessageBubble.tsx:24,49`.
- `img`/`video` `src` строится исключительно через `artifactUrl(id)` с строгой
  UUID-регуляркой `ARTIFACT_ID_RE` (`client.ts:53-54,100-104`); произвольные URL
  из ответа не используются (`MessageBubble.tsx:16-23`).
- Логирования PII/launch-параметров в консоль не найдено.

### [SEVERITY: Low] LAUNCH_PARAMS фиксируется один раз при загрузке модуля
`src/api/client.ts:51`
Значение читается на этапе импорта модуля. Для VK Mini App это приемлемо (URL не
меняется в рамках сессии), но если хост обновит параметры без перезагрузки —
клиент продолжит слать старую подпись.
**Рекомендация:** опционально — читать `window.location.search` внутри
`request()` на каждый вызов. Секреты в бандле отсутствуют (только относительные
пути и заголовки).

## 3. Утечки данных

### [Verdict: OK] Межпользовательский доступ, тексты ошибок, S3-доступ
- Cross-user закрыт ownership-проверками (см. §1). Баланс — только своего
  пользователя (`handler.go:343-367`).
- Ошибки клиенту обезличены: `unauthorized` / `not found` / `internal error`;
  причина отклонения подписи пишется только в server-side лог
  (`handler.go:109-114`). Стек/SQL/пути не утекают.
- Артефакты отдаются через бэкенд-роут `GET /miniapp/artifacts/{id}` с проверкой
  владельца и `Cache-Control: private` (`handler.go:407-424`); публичные ссылки в
  Mini App-пути не используются. Presigned-URL (`s3.go:111-118`) задействован
  только в отдельном VK-delivery-пути под TTL (`ArtifactURLTTL`).

### [SEVERITY: Low] localStorage хранит контент чатов в открытом виде
`src/chat/store.ts:4,18-24`
Ключ `vk_miniapp_chats_v1` хранит промпты и сгенерированные тексты (до 50 чатов).
Это не секреты/токены, но пользовательский контент остаётся в браузере в
plaintext (общий/публичный девайс). Запись обёрнута в `try/catch`, лимит
`MAX_CHATS=50` есть.
**Рекомендация:** задокументировать поведение; опционально — TTL/очистка или
отказ от хранения тел сообщений. Токенов/launch-params в хранилище нет — это
главное и оно соблюдено.

## 4. Утечки ресурсов

### [Verdict: OK] Бэкенд: горутины, закрытие rows, ретраи, таймауты
- Worker: разделение `readCtx`/`handlerCtx`, graceful drain с grace-таймаутом и
  принудительной отменой (`cmd/worker/main.go:61-63,296-320`).
- `rows.Close()` через `defer` во всех листингах (`job.go:134,179`,
  `billing.go:165`).
- Ретраи ограничены `MAX_ATTEMPTS` (`config.go:210`), backoff экспоненциальный.
- Внешние вызовы под таймаутом: OpenAI `http.Client{Timeout: 120s}` и модерация
  `30s` (`openai/openai.go:103`, `openai/moderation.go:43`).

### [SEVERITY: Medium] Главный useEffect сбрасывает mountedRef при смене числа чатов
`src/chat/ChatScreen.tsx:177-231`
Эффект завязан на `chats.length` (и набор колбэков). Его cleanup делает
`mountedRef.current = false` при **каждом** перезапуске (например, при создании
нового чата), а тело снова ставит `true`. Активные `poll()`-циклы читают
`mountedRef.current` и могут «замолчать» (`return`) в момент, когда компонент на
самом деле смонтирован, просто изменилось число чатов. На практике повторный
прогон возвращает `true`, но логика хрупкая: флаг «размонтирован» смешан с
«эффект перезапустился».
**Рекомендация:** держать `mountedRef` в отдельном `useEffect(() => {...}, [])`
(установка/сброс только на mount/unmount), а сидирование/баланс — в другом
эффекте; либо сузить зависимости (использовать функциональные сеттеры и убрать
`chats.length`). Сами по себе: дедуп поллеров (`pollingRef` Set), `POLL_MAX`,
остановка на терминальном статусе и `unsubscribe` у vk-bridge
(`useBridge.ts:68-71`) — реализованы корректно.

## 5. Оптимизация

### [SEVERITY: Low] Нет составного индекса под ListByUser
`migrations/000001_init_schema.up.sql:82-83`, `job.go:124-145`
Есть `jobs_user_id_idx` и `jobs_status_idx`, но `ListByUser` фильтрует по
`user_id` и сортирует `ORDER BY created_at DESC`. На больших объёмах эффективнее
составной индекс `(user_id, created_at DESC)` (убирает сортировку). Пагинация
(`LIMIT/OFFSET`) и `limit+1` для `has_more` реализованы корректно.
**Рекомендация:** добавить индекс `jobs (user_id, created_at DESC)` при росте
данных.

### [SEVERITY: Low] Затравка истории делает N запросов за текстом
`src/chat/ChatScreen.tsx:208-221`
При первом старте без локальных чатов для каждой завершённой `text_generate`
задачи вызывается `resolveBotText` → отдельный `GET /miniapp/artifacts/{id}`
(N запросов). Приемлемо при малой истории, но масштабируется линейно.
**Рекомендация:** ограничить число дозагрузок (например, только видимые/последние
K сообщений) или подгружать лениво при появлении в вьюпорте.

### [SEVERITY: Low] Поллинг с фиксированным интервалом без backoff
`src/chat/ChatScreen.tsx:22-23,86-133`
Фиксированные 2 c × до 90 итераций (3 мин) на каждую задачу; параллельные задачи
= параллельные поллеры. Иммутабельные апдейты по `id` и автоскролл только на
изменение `messages` — корректны; VKUI из бандла удалён.
**Рекомендация:** опционально — экспоненциальный/затухающий интервал для долгих
задач (видео).

## 6. Архитектура

### [Verdict: OK] Чистота слоёв и стейт-машина денег/Job
- Adapter `miniapp` зовёт сервисы (`joborchestrator`, `billingservice`) и domain;
  DTO-конвертация изолирована в `dto.go`. Провайдеры не знают о VK/биллинге.
  Инфраструктура в домен не течёт.
- Job-переходы явные и атомарные, `UpdateStatus` пинит предыдущий статус в
  `WHERE ... AND status = $from` (`job.go:84-100`), потеря конкурентного перехода
  невозможна (`ErrConflict`).
- Деньги: `Reserve` считает доступный баланс под `FOR UPDATE`
  (`billing.go:304-320`), ledger append-only с `UNIQUE idempotency_key`,
  `Capture`/`Release` проверяют `status == reserved` иначе `ErrConflict`. Двойного
  списания нет. Стартовый грант — committed ledger-запись в одной транзакции
  (`billing.go:81-119`, инвариант #14).

### [SEVERITY: Medium] Выбор модели на фронте не доходит до бэкенда
`src/chat/Composer.tsx` (UI) + `src/chat/ChatScreen.tsx:163-166` vs
`internal/adapter/inbound/miniapp/dto.go:25-32`
UI даёт выбрать модель (`modelId`) и кладёт её в локальное сообщение, но
`createJob({ operation, prompt })` отправляет на бэкенд только `operation` и
`prompt`. `CreateJobRequest` поля `model` не имеет — выбор модели чисто
косметический и игнорируется сервером (модель определяется конфигом воркера).
Рассинхрон ожиданий пользователя и поведения системы.
**Рекомендация:** либо добавить `model` в `CreateJobRequest` + проброс в
оркестратор/провайдер с валидацией по белому списку, либо убрать выбор модели из
UI до готовности контракта. Зафиксировано как известный TODO в `TASKS.md`.

### [SEVERITY: Medium] getArtifact жёстко зависит от доступности S3 в api-процессе
`cmd/api/main.go:81-92,137-149`, `handler.go:369-373`
api теперь подключается к S3 для чтения артефактов. При сбое подключения
`objectStore = nil`, и `getArtifact` отдаёт `503 artifact storage unavailable` —
тексты/картинки/видео в Mini App молча недоступны, хотя Job завершился успешно.
Это новая инфраструктурная связность api↔S3 (раньше intake-only). Деградация
тихая для пользователя.
**Рекомендация:** в проде считать S3 обязательной зависимостью для Mini App
(падать/алертить, а не `Warn`), либо явно отражать недоступность артефактов в
UI; задокументировать связность в `ARCHITECTURE.md`.
Замечание (резолв): ранее зафиксированный рассинхрон «фронт ждёт
`GET /miniapp/artifacts/{id}`, а роута/`result_text` нет» — **устранён**: роут
существует (`handler.go:75`), текст отдаётся как `text/plain` артефакт
(`artifactservice/service.go:102-103`), фронт корректно его читает
(`client.ts:107-118`). Поля `result_text` в DTO нет намеренно.

### [SEVERITY: Low] Дублирование helper-ов между adapter-пакетами
`internal/adapter/inbound/admin/handler.go:162-205` и
`internal/adapter/inbound/miniapp/handler.go:438-464`
`parsePagination`, `writeJSON`, `writeError` продублированы в admin и miniapp;
`ensureUser` повторяет логику VK-инбаунда (отмечено комментарием
`handler.go:135-136`). God-объектов нет, границы ответственности соблюдены.
**Рекомендация:** при следующем рефакторинге вынести HTTP-хелперы в общий
internal-пакет (вне «режима экономии» — не срочно).

## ТОП-5 приоритетов на исправление

1. **[High] Rate-limiting на `/miniapp/*`** (особенно `POST /jobs`) —
   `cmd/api/main.go:158`. Сейчас интейк Mini App ничем не ограничен по частоте.
2. **[Medium] Жёсткая проверка `vk_ts`** при `maxAge > 0` — `sign.go:75-86`.
   Отсутствие/битость метки не должны отключать TTL (риск replay).
3. **[Medium] Контракт «модель»** — `dto.go` / `ChatScreen.tsx`. Либо
   пробросить выбранную модель в API с валидацией, либо убрать из UI.
4. **[Medium] Зависимость api↔S3 для артефактов** — `cmd/api/main.go:88-92`.
   Сделать обязательной/наблюдаемой в проде, отразить деградацию в UI.
5. **[Low→Medium] Развязать `mountedRef` от перезапуска эффекта** —
   `ChatScreen.tsx:177-231`. Установка/сброс флага только на mount/unmount.

> Не найдено / требует уточнения: модель развёртывания фронта (same-origin proxy
> vs прямой доступ) для решения по CORS; политика retention/шифрования контента в
> localStorage. Эти пункты не подтверждаются кодом и не додумывались.
